OurTrips used to assume the planning conversation happened somewhere else. You would plan in an external agent, connect OurTrips, and send the result over.

That still works, but it is no longer the main path.

Now OurTrips creates the trip from the beginning. The product has become a travel planning surface, not only a place to store a finished itinerary.

## The intake gives the agent a frame

The flow starts with the travel basics: destination, dates, travelers, origin, budget, pace, must-dos, and known bookings.

Those answers matter because they define the shape of the trip. A relaxed anniversary weekend, a family rail route, and a two-week expedition should not be planned the same way.

## References keep the plan grounded

After the basics, you can add the material that normally gets scattered:

- Booking details
- Hotel options
- Notes and preferences
- Restaurant ideas
- Old itineraries
- PDFs and reference material
- Open questions

The agent uses that context to build a plan from your actual constraints instead of guessing from a generic destination prompt.

## The first draft is a real guide

When the first draft is ready, it opens as an OurTrips guide. It is not just text. It has days, places, stays, meals, transport, route points, maps, practical tips, and visible open decisions.

That matters because the plan is immediately portable. You can share it. You can inspect it on mobile. You can see whether the day flow makes sense.

The output should help you plan and help you travel.

## The chat edits the itinerary

The built-in chat stays with the trip. That changes the feel of AI planning.

Instead of asking an agent for another free-floating answer, you can ask for changes to the actual guide:

> Add a slower morning after arrival.

> Replace the second hotel with the one we booked.

> Find a dinner near the stay and note whether reservations are needed.

> Make the route less rushed.

The agent can make targeted edits, keep source notes aligned, and review related logistics when a change affects more than one day.

## The guide is built for the road

OurTrips also includes the travel-day details that matter later: day-by-day access, maps, share links, offline saving, booking status, and practical notes.

That is the reason the product is not just a chat interface. Travel needs a durable format. The plan has to survive the moment when you close the laptop and open your phone in a new city.

## External agents are optional now

If you already like planning in Claude, Codex, or another remote-MCP-capable agent, the connector still works at:

```copy
https://ourtrips.to/mcp
```

But you do not need it to start.

The primary OurTrips flow is now: collect the information, create the trip, refine the guide, and carry it with you.
